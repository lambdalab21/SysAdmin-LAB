# Shotts Chapter 25: Starting a Project

This guide is for Chapter 25, **Starting a Project**, from William Shotts's *The Linux Command Line*.

The goal is not to type an HTML-report script mindlessly. The goal is to learn how a script grows from a tiny working version into a useful project.

Core habit:

```text
small working version
→ inspect output
→ explain every line
→ improve one small part
→ test again
```

Feynman analogy:

```text
A script project is like building a small model house.
You do not begin with plumbing, paint, furniture, and wiring all at once.
You first build a simple frame that stands.
Then you add one feature at a time.
```

Working directory:

```bash
mkdir -p ~/tlcl-ch25-project
cd ~/tlcl-ch25-project
```

Daily discipline:

```text
Before typing:
  What am I trying to make the script do?
  What output do I expect?
  Which line is responsible for which part?

After running:
  Did it run?
  Did it produce the expected output?
  Can I explain every line?
  What is the next smallest improvement?
```

# Before reading: Feynman preview

Explain this idea before reading:

```text
If I want to build a report generator, the first goal is not a beautiful report.
The first goal is a script that creates a valid tiny report.
```

```text
I can test it.
I can improve it.

```

# After reading: concept questions

Answer without looking back:

1. Why does Shotts begin with a minimal document?
2. What is the advantage of a script that runs but does very little?
3. What does it mean to redirect script output into a file?
4. Why is stdout useful for report generation?
5. Why should the first version avoid cleverness?

Do not do the exercise until these are answered.

# Exercise 1: Create the smallest report script

## Skill being trained

He is learning to make a runnable starting point.

## Predict before typing

Answer:

```text
What file will I create?
What will the script print?
Where will the output go when I redirect it?
What command will I use to inspect the result?
```

## Create the script

```bash
cd ~/tlcl-ch25-project

cat > sys_info_page.sh <<'EOF'
#!/usr/bin/env bash

# sys_info_page.sh - generate a simple system information page

echo "<html>"
echo "<head>"
echo "<title>System Information Report</title>"
echo "</head>"
echo "<body>"
echo "<h1>System Information Report</h1>"
echo "</body>"
echo "</html>"
EOF
```

## Run it first with Bash

```bash
bash sys_info_page.sh > sys_info_page.html
```

Inspect:

```bash
cat sys_info_page.html
```

## Explain-back

Explain every line:

```text
#!/usr/bin/env bash:
comment line:
each echo line:
> sys_info_page.html:
```

# Exercise 2: Check that the script still works after a small change

## Skill being trained

He is learning to change one thing and test.

## Predict before typing

Answer:

```text
What one line will I add?
Where should it appear in the output?
How will I verify it appeared?
```

## Add one line

Edit the script and add this line after the `<h1>` line:

```bash
echo "<p>Generated by a Bash script.</p>"
```

Run:

```bash
bash sys_info_page.sh > sys_info_page.html
grep "Generated" sys_info_page.html
```

## Explain-back

Answer:

1. Did the script still run?
2. Did the new line appear where expected?
3. Why is one small change safer than many changes at once?
4. What would you do if the output was wrong?
